# AskUserQuestion 작성 원칙

> 역할: 슬래시 커맨드·에이전트가 `AskUserQuestion`으로 사용자 결정을 받을 때 따르는 작성 규약.
> 대상: `/new-feature`, `/research`, `/write-requirements`, `/design-review`, `/plan-phase` 등 `AskUserQuestion`을 호출하는 모든 슬래시 커맨드와 에이전트.

기준: 비전문가 사용자가 5초 안에 옵션을 고를 수 있어야 한다.

## 원칙

- **질문 텍스트는 일상어로**: 한 줄 의문문. 정의 없이 약어를 첫 문장에 박지 않는다.
- **옵션 label은 의사결정 단위**: "권장 / 더 엄격 / 더 가볍게" 또는 "지금 / 나중 / 안 함" 같이 사용자가 무엇을 고르는지 한눈에 보이게. 내부 약어를 label에 넣지 않는다.
- **프로젝트 용어는 괄호 풀이**: 옵션 본문에 ADR 번호, 내부 jargon, `vertical slice`, `cascading`, `headless`, `tr_key` 같은 내부 용어를 처음 쓸 때는 `pillar(핵심 지표 한 축)` 형태로 한 줄 풀이를 붙인다.
- **description은 영향 중심**: "무엇을 얻고 무엇을 잃는가"를 1~2줄. 완료까지 걸리는 노력, 추가 수동 QA 항목, 후속 phase 영향 정도를 보여준다.
- **긴 옵션은 요약 → 디테일 순서**: 옵션 본문이 두 줄을 넘으면 첫 줄에 핵심 요약("기본 4건 그대로"), 두 번째 줄 이후에 세부 항목.
- **결정에 필요한 만큼만**: 7개짜리 기술 체크리스트를 옵션 안에 넣지 않는다. 필요하면 본문에서 한 줄로 설명하고 "기본 / 더 자세히 / 직접 작성"으로 분기한다.
- **추천은 첫 번째, `(권장)` 표시**: 권고가 있다면 첫 옵션 label 끝에 `(권장)`.
- **최대 4 옵션, 4 질문**: 도구 제약이자 인지 부하 제약. 더 많아지면 단계로 쪼갠다.

## 나쁜 예 / 좋은 예

```text
# 나쁨 — 첫 줄부터 내부 용어, 옵션 본문이 기술 체크리스트
질문: Acceptance Criteria — 'vertical slice 완료' 조건 목록?
옵션 A: (권장) brief Acceptance shape 그대로 (4건)
        (1) hunger pillar cascading 3 manual QA 재현
        (2) temperature pillar cascading 3 재현 …
옵션 B: 권장 + 기술 기준 추가 — 4건 + (5) save/load 후 상태 복원
        (6) headless 프레임 예산 내 (pillar당 0.5ms) (7) UI rebuild 주기 ADR-011 기준 내 …

# 좋음 — 일상어 + 영향 중심 + 디테일은 본문 1줄
질문: '완료'로 인정할 조건은 어느 수준이면 충분할까?
옵션 A: 기획서에 적힌 4가지 그대로 (권장)
        description: 핵심 시나리오 3개 + 통합 trade-off 1개. 수동 QA 4건.
옵션 B: 기획서 4건 + 기술 안정성 4건
        description: 위 4건에 저장/불러오기 정상, 프레임 성능, UI 갱신, 한·영 텍스트 등록 확인 추가. 수동 QA가 약 2배.
옵션 C: 핵심만 5건으로 압축
        description: 각 시나리오·통합 trade-off·저장 복구·성능·텍스트 한 줄씩. QA 부담 가장 적음.
```

옵션을 짜기 전 자문: "비전문가 사용자가 이 옵션을 읽고 어느 쪽이 자기 의도와 맞는지 바로 알 수 있나?" 답이 "아니오"면 풀어 쓴다.
