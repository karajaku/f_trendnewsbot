# daily_digest V1 — Stage 0 Discovery Research

> 역할: V1 브리프 작성 전에 (1) 외부 매일 뉴스 다이제스트 제품 디자인 레퍼런스와 (2) 현재 코드베이스의 유사 시스템 유무를 가볍게 조사한다. 브리프가 "정보 없는 종이"가 아니라 현실 기반으로 작성되도록 한다.
> 대상: `daily_digest_v1-brief.md` 작성자 (Stage 1 진입자), 회사 도메인 컨텍스트를 이해한 후 브리프 시사점을 인용하려는 사람.

---

## 1. 조사 질문 (사용자 서명)

운영체제 부트스트랩 직후 연속 문서 작성 흐름에서, 사용자가 "리서치부터 순차적으로 진행"을 명시함(2026-05-19). 별도 AskUserQuestion 없이 PRD·CLAUDE.md·회사 소개 문서에서 도출한 핵심 5개 질문으로 진행. 사용자 마지막 일괄 검토 단계에서 이 질문 목록을 보강·수정할 수 있다.

1. **포맷 — 사내 임직원이 출근 직후 5분 안에 트렌드를 파악하는 매일 뉴스 다이제스트는 어떤 본문 구조와 길이가 표준인가?**
2. **카테고리 — 한 다이제스트 안에서 "AI 트렌드 / 농산물 유통 / 회사 관심 키워드" 3축이 시각적·내용적으로 어떻게 분리되어야 직원이 빠르게 흩어 읽을 수 있는가?**
3. **요약 길이와 신뢰성 — LLM 요약을 직원에게 보낼 때 hallucination을 가장 작게 만드는 본문 길이·구성은? 원문 링크 동봉 외에 어떤 안전장치가 있나?**
4. **시간 표기 — 매일 발송 다이제스트에서 "오늘", "어제", 발행 시각, 발송 시각을 어떻게 표기해야 한국인 사용자가 자연스럽게 받아들이는가?**
5. **코드베이스 — 현재 저장소에 이 다이제스트와 유사한 봇·스크립트·자동화가 이미 존재하는가? 이식할 만한 helper나 회사 도메인 데이터(키워드·산지 목록)가 있는가?**

## 2. 디자인 레퍼런스

> 사용자가 외부 조사를 명시적으로 허용한 단계가 아니므로, 본 리서치는 LLM의 일반 지식(2026-01 cutoff 기준 공개 정보)에서 도출한 디자인 패턴만 정리한다. URL·retrieved 날짜는 별도 외부 조사 사이클(향후)에서 추가한다.

### 2-1. 매일 뉴스 다이제스트 제품군 — 공통 디자인 패턴

| 제품군 | 포맷 패턴 | 본문 길이 | 카테고리 처리 |
|---|---|---|---|
| **Morning Brew / Axios PM 등 영문 비즈니스 다이제스트** | 카테고리 헤더 + 항목별 "제목 → 한 문장 요약 → why it matters 한 줄 → 원문 링크" | 5~10분 읽기 분량 | 굵은 카테고리 헤더로 분리, 각 카테고리 3~5건 |
| **TLDR / TLDR AI / Benedict's Newsletter — IT/AI 큐레이션** | 항목별 "제목 + 2~3 문장 요약 + 원문 링크"의 단순 리스트 | 3~5분 분량 | 카테고리 없거나 1~2개로 단순화 |
| **국내 뉴스레터 — 미라클레터(머니투데이), 어피티 등 한국어 다이제스트** | 인사말 → 오늘의 주제 묶음 → 항목별 "이모지 헤더 + 한국어 요약 + 원문 링크" | 한국어로 5~7분 | 주제 묶음당 2~4건, 헤더에 카테고리 아이콘 |
| **Google Alerts / 기업 사내 클리핑 서비스** | 키워드 매칭 결과를 그대로 나열 (요약 없거나 매우 짧음) | 분량 가변 | 키워드별 그루핑 |

**공통 관찰**:

- 직원이 "출근 직후 5분 안에 읽기" 같은 짧은 시간 제약을 가진 사내 다이제스트라면, **카테고리당 5~10건 + 항목당 한 줄~두 줄 요약**이 보편적이다. PRD V1 명시 범위와 일치.
- 항목 앞에 **별표·이모지·번호** 같은 시각적 앵커가 있으면 흩어 읽기가 빨라진다. 한국어 환경에서 이모지 사용은 회사 톤에 맞춰야 한다(팜보스 사내 톤은 미확인).
- "**Why it matters**" 같은 한 줄 코멘트가 있으면 직원이 클릭 여부를 빠르게 결정한다. 단 V1에서 시행하면 LLM이 사실을 더 만들어내야 하므로 hallucination 위험이 커진다 — V2 검토 항목으로 분리 권장.

### 2-2. LLM 기반 요약 안전 패턴 — 일반 공개 자료에서 도출

- **원문 링크 동봉**은 최소한의 안전장치다. LLM 요약 옆에 항상 원문 URL을 노출하면 직원이 의심 시 즉시 검증 가능(CLAUDE.md CRITICAL: 요약은 보조, 원문 링크가 일차 정보 — 이미 명시).
- **요약 길이는 짧을수록 hallucination 위험이 작다.** 3 문장 이상으로 늘리면 "원문에 없는 인과·수치"를 LLM이 만들어낼 확률이 올라간다. V1은 2 문장 이내로 시작 권장.
- **숫자·고유명사·날짜**는 원문에서 그대로 인용하도록 프롬프트로 강제한다. "원문에 명시되지 않은 수치는 생성하지 말 것"을 시스템 프롬프트에 박아둔다.
- **카테고리 점수화는 별도 호출로 분리**하는 패턴이 흔하다. 한 호출에서 점수+요약을 동시에 요구하면 점수가 요약 결과에 끌려가는 결합이 생긴다. 호출 분리 시 비용은 늘지만 quality가 좋아진다 — V1 비용 hard cap(월 $20) 안에서 trade-off 검토 필요.

### 2-3. 한국어 시간 표기 관례

- 한국어 다이제스트는 본문에 "**5월 19일(월) 오전 7:30 KST 발송**" 같은 *명시적 한국 날짜·요일·시간대* 표기가 일반적.
- 기사 발행 시각은 "*N시간 전*" 같은 상대 시각보다 "*5월 19일 06:12 KST*" 같은 절대 시각이 정정 가능성·신뢰감 측면에서 낫다.
- "*오늘*", "*어제*"는 발송 KST 기준으로 일관 적용해야 직원 혼동이 없다. UTC 환산 실수가 흔한 영역.

## 4. 결론 — 브리프 반영 시사점

브리프(Stage 1) 본문 작성 시 다음 5개 시사점을 인용한다. 각 결론은 PRD·CLAUDE.md와 충돌하지 않는 범위에서 도출됐다.

1. **본문 포맷은 "카테고리 헤더 + 항목 5~10건 + 항목당 제목·한국어 요약 2 문장 이내·원문 링크"** 구조로 시작한다. Morning Brew/TLDR 계열의 검증된 패턴이며 직원이 5분 안에 흩어 읽기에 부합. (디자인 레퍼런스 §2-1)
2. **요약 신뢰성은 "짧게 + 원문 링크 + 숫자/고유명사 인용 강제"** 3축으로 통제한다. V1에서 "why it matters" 같은 LLM 코멘트는 도입하지 않는다 — hallucination 위험이 가치보다 크다. (§2-2)
3. **점수화와 요약은 호출을 분리한다.** 비용 hard cap 안에서 가능한지는 Stage 3 Tech Research에서 토큰 추정으로 확정한다. 분리하지 못하면 V1은 "단일 호출 + 짧은 요약"으로 시작하고 V2에서 분리. (§2-2)
4. **모든 시각 표기는 KST 절대 시각**으로 통일한다. "오늘/어제"는 사용을 최소화한다. cron의 UTC 환산은 매 라인 주석으로 박는다(CLAUDE.md CRITICAL과 일치). (§2-3)
5. **"팜보스 관심 키워드" 카테고리의 초기 시드는 [docs/팜보스_회사소개.md](docs/팜보스_회사소개.md)에서 추출한 산지·작목·법인명**으로 시작하고, 직원 피드백 1주일 dry-run 후 보강한다. 유사 사내 봇 선례 유무는 사용자 확인 필요. (§3)

## 5. 미결 항목 (Stage 1 또는 사용자 일괄 검토 시 확인)

- 회사 내부에 매일 뉴스 클리핑·자동 메일러·산지 시세 알림 같은 **선례가 있었는지** — 있으면 톤·수신자 운영 관행을 참고할 수 있음.
- 직원이 이미 구독 중인 **외부 뉴스레터**(미라클레터·TLDR 등)와의 중복 회피 정책 — V1 발송이 직원 메일함을 시끄럽게 만들지 않을지.
- 한국어 다이제스트 톤에서 **이모지 사용 가부** — 회사 톤이 정중·간결인지 캐주얼인지 확인 필요.
